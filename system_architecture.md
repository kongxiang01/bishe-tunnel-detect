# 隧道交通综合监控系统架构文档（毕设 MVP 版）

## 1. 项目目标

本系统用于毕业设计，目标是实现可演示、可运行、可答辩的最小可用产品（MVP），不追求生产级复杂能力。

MVP 只解决三个核心问题：
1. 能看：实时查看隧道监控视频。
2. 能识别：识别车辆并统计基础交通参数。
3. 能告警：对少量典型异常事件进行预警和记录。

## 2. MVP 范围定义

### 2.1 必做功能（答辩核心）

1. 单路或少量视频流接入（建议 1-4 路 RTSP）。
2. 车辆检测与跟踪（YOLO + DeepSORT 基础实现）。
3. 交通参数统计（车流量、平均速度）。
4. 异常事件告警（建议先做 2 类：超速、违停）。
5. Web 可视化页面（实时画面、告警列表、历史查询）。
6. 数据持久化（事件记录与统计结果落库）。

### 2.2 非必做功能（可选加分）

1. 多级权限管理（仅保留简单登录即可）。
2. 多机集群与高可用部署。
3. 对象存储与复杂冷热分层。
4. 外部平台对接（交警平台、短信平台）。

## 3. 简化技术架构

采用前后端分离，但部署保持简化：单机部署优先。

### 3.1 架构分层

1. 前端层：React + Ant Design + ECharts。
2. 后端层：Spring Boot（REST API + WebSocket）。
3. 算法层：Python 推理服务（YOLO + DeepSORT）。
4. 数据层：MySQL（可选 Redis，不强制）。

### 3.2 简化数据流

1. 摄像头视频输入算法服务。
2. 算法输出检测框、轨迹、速度、事件。
3. 后端接收结果并写入 MySQL。
4. 前端通过接口和 WebSocket 展示实时信息与历史记录。

## 4. 模块设计（MVP 粒度）

### 4.1 视频与推理模块

1. 使用 OpenCV/FFmpeg 读取 RTSP。
2. 按固定帧率抽帧推理（建议 5-10 fps）。
3. 输出字段：时间戳、车辆 ID、类别、速度、事件类型。

### 4.2 事件规则模块

1. 超速：速度大于阈值触发。
2. 违停：目标在 ROI 区域内静止超过阈值时间触发。
3. 事件去重：同一目标短时间内不重复告警。

### 4.3 后端业务模块

1. 设备管理：仅支持基础增删改查。
2. 事件管理：告警入库、分页查询、按时间筛选。
3. 统计管理：按分钟或小时聚合车流量和平均速度。

### 4.4 前端展示模块

1. 实时监控页：视频画面 + 实时计数。
2. 告警页：告警列表 + 告警详情。
3. 历史统计页：折线图/柱状图展示流量和速度。

## 5. 数据库最小设计

建议只保留 4 张核心表：
1. user：用户信息（用于登录）。
2. device：摄像头信息。
3. event：告警事件（类型、时间、截图路径、状态）。
4. traffic_stat：交通统计（时间窗口、车流量、平均速度）。

说明：MVP 阶段不引入复杂分库分表。

## 6. 部署方案（毕设推荐）

### 6.1 单机部署

1. 一台 PC 或工作站部署前端、后端、算法服务和 MySQL。
2. 若有 NVIDIA GPU，优先使用 GPU 推理；否则降低帧率保证可运行。

### 6.2 推荐最低配置

1. CPU：6 核以上。
2. 内存：16GB 以上。
3. GPU：可选（6GB 显存以上更好）。
4. 磁盘：100GB 以上。

## 7. 验收指标（按毕设可实现设定）

1. 实时性：单路视频端到端延迟 <= 3 秒。
2. 并发：稳定支持 1-4 路视频（根据硬件实际说明）。
3. 功能完整：必做功能全部可演示。
4. 稳定性：连续运行 2 小时无崩溃。
5. 精度说明：在自建测试集上给出准确率/召回率，不强求工业级指标。

## 8. 开发计划（8 周示例）

1. 第 1-2 周：需求冻结、数据准备、基础模型验证。
2. 第 3-4 周：推理服务与事件规则开发。
3. 第 5-6 周：后端接口与数据库开发。
4. 第 7 周：前端页面联调与功能完善。
5. 第 8 周：测试、文档、答辩演示材料整理。

## 9. 可行性评估（MVP）

### 9.1 技术可行性

结论：高可行。

原因：
1. 技术栈成熟，教程与开源实现丰富。
2. MVP 只做 1-4 路视频和 2 类事件，复杂度可控。
3. 单机部署降低系统工程难度。

### 9.2 时间可行性

结论：中高可行。

前提：
1. 严格控制范围，不临时增加复杂需求。
2. 优先打通主链路（采集 -> 推理 -> 告警 -> 展示）。

### 9.3 风险与应对

1. 风险：模型精度不稳定。
   应对：先用预训练模型，优先保证可运行，再小规模调参。
2. 风险：硬件性能不足。
   应对：降低输入分辨率和推理帧率，确保实时演示。
3. 风险：联调时间不足。
   应对：先实现离线回放演示，再补实时流演示。

## 10. 答辩建议

1. 强调“为什么是 MVP”：目标是验证方案有效性，而非工业级商用系统。
2. 准备 1 条完整演示链路：视频输入 -> 检测跟踪 -> 告警弹出 -> 历史查询。
3. 展示对比数据：开启/关闭告警规则前后的效果差异。
4. 明确后续扩展点：多路并发、更多事件类型、集群部署。

## 11. 结论

该方案已按毕设目标进行简化，功能闭环清晰、实现路径明确、工作量可控。只要按 MVP 范围执行，具备较高概率按期完成并通过毕业设计答辩。
