# 毕业设计项目开发路线指南 (Project Roadmap)

**核心理念：优先实现 MVP（最小可行性产品）闭环贯通，避免陷入早期的过度开发与过度设计。**

为了确保本次毕业设计能够平稳、高质量地推进，特别制定本开发路线图。本路线分为三个阶段，建议严格按顺序执行，确保每个阶段均有“可交付、可演示”的成果。

---

## 阶段一：打通全栈数据链路（当前最高优先级）
**目标**：不修改算法内部细节，只利用现有官方模型跑通前端、后端与算法端的交互流程。

1. **算法端 (Python/FastAPI)**：
   - [x] 配置完成专属 Conda 环境 (`grad_algo`)。
   - [x] 拉取 YOLOv5 源码并跑通基础的 `detect.py`。
   - [x] 将 YOLOv5 集成至 FastAPI，开放 `/infer` HTTP POST 接口。
   - [x] 采用最极速的“传递本地图片绝对路径”方案，返回图像中标注框 (`bbox`) 和类别坐标。
2. **后端服务 (Java/Spring Boot)**：
   - 编写调用本计算节点（Python 端 `/infer` 接口）的 HTTP Client（如 `RestTemplate` 或 `WebClient`）。
   - 实现业务逻辑：控制系统按帧抓取本地图片或摄像流，发给算法端，并将处理好的告警数据持久化写入数据库。
3. **前端展示 (React/Vite)**：
   - 编写数据监控大屏页面。
   - 根据后端下发的带有 `bbox` 坐标的 JSON 数据，使用 HTML5 Canvas 或是原生 UI 组件将检测框实时叠加在视频流或图片流上。

*验收标准：前端页面上能够流畅地播放一段测试视频（或图片序列），各种车辆和行人身上成功带上 YOLO 识别的红框。*

---

## 阶段二：引入追踪与业务逻辑 (多目标追踪 Tracker)
**目标**：实现“轨迹与计算”，从“单纯抓拍”升级为“视频动态监测”。

1. **引入 DeepSORT / ByteTrack**：
   - 原生 YOLO 无法记住上一帧的车和下一帧的车是否是同一辆。需要在 `algorithm` 端集成目标追踪算法。
   - 给每一个进入画面的车辆赋予唯一的 `track_id`。
2. **完成特定交通业务需求**：
   - **单帧分析**：车辆统计、车流密度。
   - **多帧推演与测速**：设定画面基准线，通过帧率及车轮位移计算车辆近似速度（如：超速预警）。
   - **轨迹异常检测**：违规变道、逆行、异常停车（长时间 `bbox` 不发生位移）。
3. **业务闭环**：当检测到交通超速异常时，产生 Event 发送给 Java 后端，后者通过 WebSocket 推送至前端大屏进行警报。

*验收标准：点击某一辆车能看到它的持续轨迹线与编号，且发生违规时报警台产生相应的预警弹窗。*

---

## 阶段三：算法精修与自定义私有模型（锦上添花）
**目标**：提升算法针对性，训练特定场景下的新模型，提升毕业设计的学术深度。

在所有系统链路全部运行顺畅且稳定之后，如有余力，再进行底层算法指标的提升：
1. **收集专属数据**：针对隧道内部环境（光线较暗）等场景收集真实的监控截图。
2. **处理长尾类别**：如果毕设含有特定的识别任务（如：隧道掉落物、火灾烟雾、特定隧道裂缝、消防器材），使用 `labelImg` 或 `Roboflow` 手动打标签。
3. **模型训练**：基于 YOLOv5 源码运行 `train.py`，经历数小时或数天的显卡训练，产出 `custom_tunnel_yolov5.pt` 权重文件。
4. **无缝替换**：由于我们的 FastAPI 做了良好的解耦，只需在 `app/main.py` 里修改加载的文件名，即可瞬间完成系统升级，一行外部代码都不需要修改。

*验收标准：产出完整的 mAP / PR 等模型评估对比图表，直接用在最终的毕业论文中水字数与撑厚度。*
